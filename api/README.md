# machorale REST API — Conventions & Contract

The JSON API the Android/mobile app consumes. Lives under `/api/v1/`, outside
Django's `i18n_patterns` (no `/fr/` `/en/` locale prefix). The existing web app,
models, and business logic are untouched — this is a JSON interface over them.

## Base URL

```
https://<host>/api/v1/
```

Everything is versioned. `v1` is frozen once the app ships — never break it.

## Authentication

JWT (djangorestframework-simplejwt). Send the access token on every protected
request:

```
Authorization: Bearer <access_token>
```

- Access token lifetime: **30 min**
- Refresh token lifetime: **14 days**, rotated on use (`ROTATE_REFRESH_TOKENS`)
- Default permission is `IsAuthenticated`. Public routes opt out explicitly
  (e.g. `ping`, OTP request/verify).

## Casing

JSON keys are `snake_case` (DRF default), matching the Django models.

## Error format

Every error raised **inside a DRF view** returns this envelope:

```json
{
  "detail": "<human-readable message>",
  "errors": { "<field>": ["<msg>", ...] }
}
```

- `errors` is `{}` for non-validation errors (auth 401, permission 403,
  404, throttle 429, 405).
- Validation errors (400) populate `errors` per field; `non_field_errors`
  holds form-level messages.
- Unmatched `/api/` paths also return this envelope (JSON 404), never an HTML
  page — see `api.views.api_not_found`.

## Pagination

Page-number pagination on all list endpoints:

```json
{
  "count": 123,
  "next": "https://.../api/v1/...?page=3",
  "previous": "https://.../api/v1/...?page=1",
  "results": [ ... ]
}
```

- Default page size: **20**. Client override: `?page_size=N` (max **100**).
- Page selector: `?page=N`.

## Throttling

Default DRF rates: `anon` 60/min, `user` 120/min. Sensitive endpoints (auth
OTP request/verify) carry stricter `django-ratelimit` limits — see each sprint.

## CORS

`CORS_URLS_REGEX = ^/api/.*$` — only API routes send CORS headers. Allowed
origins via `CORS_ALLOWED_ORIGINS` env var (comma-separated). Native mobile
clients don't need CORS; this is for browser-based clients.

## Money

Always `Decimal`, never `float`. Currency is XAF.

## Endpoints

### Sprint 0 — foundations
| Method | Path            | Auth | Description                    |
|--------|-----------------|------|--------------------------------|
| GET    | `/api/v1/ping/` | none | Liveness probe → `{"status":"ok"}` |

### Sprint 1 — authentication (OTP-email → JWT)
| Method | Path                       | Auth   | Description |
|--------|----------------------------|--------|-------------|
| POST   | `/api/v1/auth/otp/request/`| none   | Body `{email}`. Triggers OTP generation + email (Celery). **Always 200** (no email enumeration). Rate-limited `5/m` per email. |
| POST   | `/api/v1/auth/otp/verify/` | none   | Body `{email, code}`. Verifies via existing `OtpCode` logic; on success marks the user verified and returns `{access, refresh, user}`. Rate-limited `5/m` per email. |
| POST   | `/api/v1/auth/refresh/`    | none*  | Body `{refresh}` → `{access, refresh}` (refresh rotated). *The refresh token is the credential. |
| GET    | `/api/v1/auth/me/`         | Bearer | Current user + profile. |

**OTP request/verify bodies**
```json
// request
{ "email": "singer@example.com" }
// verify
{ "email": "singer@example.com", "code": "12345" }
// verify success →
{ "access": "<jwt>", "refresh": "<jwt>",
  "user": { "id": 1, "username": "...", "email": "...",
            "first_name": "", "last_name": "", "is_verify": true,
            "profile": { ... } } }
```

The OTP code is 5 digits, single-use, valid 10 minutes — generated by the same
`OtpCode` model the web flow uses. Re-requesting reuses the user's row
(`get_or_create`) and regenerates the code.

### Sprint 2 — chorale context & dashboard
| Method | Path                                  | Auth   | Description |
|--------|---------------------------------------|--------|-------------|
| GET    | `/api/v1/chorales/`                   | Bearer | Chorales the caller belongs to, each with the caller's `role` + `is_admin`. Unpaginated (small set). |
| GET    | `/api/v1/chorales/<slug>/dashboard/`  | member | Cached dashboard aggregate (reuses `get_dashboard_stats`). |

Permission `IsChoraleMember` (in `api/v1/permissions.py`) mirrors
`ChoraleRequireMixin`: resolves the chorale from `<slug>`, requires a
`Membership`, and attaches `request.chorale` / `request.membership`. Unknown
slug → **404**; authenticated non-member → **403**. Role-gated variants
(`IsSecretaryOrAdminOrReadOnly`, `IsTreasurerOrAdminOrReadOnly`,
`IsCensorOrAdminOrReadOnly`, `IsChoraleAdminOrReadOnly`) build on it for later
sprints — read open to any member, unsafe verbs gated by role (`is_admin`
always bypasses).

**Dashboard shape** (money as Decimal strings):
```json
{
  "total_members": 12,
  "upcoming_event_count": 3,
  "cash_balance": "150000.00",
  "cash_in_total": "200000.00",
  "cash_out_total": "50000.00",
  "contributions_collected_this_month": "75000.00",
  "active_contribution_count": 2,
  "open_sanctions_count": 1,
  "unjustified_absences_this_month": 4
}
```

### Sprint 3 — members
| Method | Path                                       | Auth          | Description |
|--------|--------------------------------------------|---------------|-------------|
| GET    | `/api/v1/chorales/<slug>/members/`         | member        | Paginated. `?search=` (name/username/email), `?role=`. |
| POST   | `/api/v1/chorales/<slug>/members/`         | secretary/admin | Create/invite. Body `{email, first_name, last_name, contact_phone?, role?}`. |
| GET    | `/api/v1/chorales/<slug>/members/<id>/`    | member        | Member detail (`id` = membership id). |
| PATCH  | `/api/v1/chorales/<slug>/members/<id>/`    | secretary/admin | Edit `role` (elevation admin-only), `first_name`, `last_name`, `contact_phone`. |
| DELETE | `/api/v1/chorales/<slug>/members/<id>/`    | secretary/admin | Remove from chorale (deletes the membership). |

- A "member" is a `Membership` joined to its `CustomUser` + `Profile`; `id` is
  the membership id.
- **Role assignment:** roles are `member|secretary|treasurer|censor` (the API
  never assigns `admin` — that's the `is_admin` flag). A non-admin writer
  (secretary) may only assign `member`; elevation requires admin → else 400.
- Create provisions `CustomUser` (unusable password — members log in via OTP) +
  `Profile` + `Membership` in one transaction, logs a `person_add` activity
  event. Duplicate email → 400.
- DELETE logs `person_remove`. The **admin membership can't be removed** (403),
  nor can you remove **yourself** (400).

## Changelog
- **Sprint 0** — API layer stood up: DRF + JWT + CORS configured, error
  envelope, pagination, `ping`, JSON catch-all 404, conventions doc.
- **Sprint 1** — Authentication: `auth/otp/request`, `auth/otp/verify`
  (issues JWT pair + user), `auth/refresh`, `auth/me`. Reuses the existing
  `OtpCode` email flow; OTP endpoints rate-limited `5/m` per email; verify is
  single-use and enumeration-safe.
- **Sprint 2** — Chorale context: `chorales/` (caller's chorales + role),
  `chorales/<slug>/dashboard/` (cached stats). New `IsChoraleMember` permission
  + role-gated variants mirroring `ChoraleRequireMixin`.
- **Sprint 3** — Members CRUD under `chorales/<slug>/members/`: list (search +
  role filter), detail, create/invite, edit, remove. Secretary/admin writes;
  role elevation admin-only; `person_add` / `person_remove` activity logged.
